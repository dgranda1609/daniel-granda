Everyone's using Claude to write code. That's not wrong — it's exceptional at that. But I've found the higher-value application is upstream: **design thinking**, before a single line of code is written or a pixel is placed.

Here's what I mean, and what I've learned from months of using it this way.

## The Problem With "AI For Design"

Most designers approach AI with the wrong frame. They treat it as an accelerant for decisions already made. "Generate me a button component." "Write the copy for this hero." That works. It's faster. But it doesn't change the quality ceiling.

The ceiling-breaking use is different: **AI as a thinking partner who challenges your assumptions before you've committed to them.**

When I started building this portfolio, I didn't open Figma. I opened a Claude conversation and described what I was trying to communicate — the specific positioning, the emotional register, the jobs I wanted the site to do. Claude pushed back on every assumption. Why serif for the headlines? Why red as the accent? Why 0px radius instead of something softer?

Those challenges forced me to articulate *why*, not just *what*. That clarity made every subsequent design decision faster and more defensible.

## Rapid Visual Conceptualization

The workflow that's changed my motion design process most is what I call "concept sprinting" with Claude.

Before building anything in After Effects or Premiere, I describe a visual concept in detail and ask Claude to:
1. Identify the strongest emotional or visual tension in the concept
2. Propose 3 alternate visual metaphors that could carry the same idea
3. Flag any clichés in my original approach and suggest unexpected substitutes

For a recent brand film for Kreyol Essence, this process surfaced a visual metaphor — botanical geometry — that we hadn't considered. It became the central visual language of the piece.

That's not code generation. That's the work that happens before tools open.

## Claude Code → Figma: The 2026 Design Stack

The convergence that matters for creative technologists: Claude Code can now push production React components directly into Figma via MCP. You build the component with correct brand tokens, correct typography, correct spacing — and the rendered state translates to editable Figma layers.

This inverts the traditional design-to-dev handoff. Design *follows* code, not the other way around. For solo creators or small studios, this eliminates an entire category of friction: the gap between what you designed and what actually got built.

I've used this to build rapid prototypes that function exactly like production interfaces, then use those prototypes in client presentations. The prototype *is* the code. No translation layer. No "but it looked different in Figma."

## What Claude Can't Do (And Why That's Important)

Claude cannot feel whether a design is *right*. It doesn't have the intuition that comes from watching someone interact with an interface for the first time, or the experience of presenting a visual direction to a skeptical creative director.

When I asked Claude to evaluate whether my portfolio's hero section "felt cinematic," the response was analytically accurate and creatively inert. It identified elements that *should* create a cinematic effect. It couldn't feel whether they actually did.

That gap — between analytical correctness and felt rightness — is where craft lives. AI accelerates the path to that gap. Getting through the gap is still human work.

## My Actual Workflow

1. **Concept phase**: Claude conversation to stress-test assumptions, generate alternative metaphors, surface clichés
2. **Visual language**: Describe palette, typography, motion principles — Claude flags inconsistencies against described brand personality
3. **Component prototyping**: Claude Code to build working UI components
4. **Figma sync**: Push to Figma for client-facing presentation
5. **Polish pass**: Human judgment on felt correctness, micro-interactions, emotional register

The AI handles everything in that stack except step 5. Step 5 is everything.
